\section*{Abstract}
% (150--250 words)

Environmental monitoring in remote, off-grid locations faces a critical ``data latency gap.'' Traditional systems force a compromise between high bandwidth, which rapidly depletes batteries, and long-range, low-power communication, which cannot support heavy payloads like image data. This project addresses this ``Bandwidth-Range-Power Trilemma'' with a hybrid IoT mesh network built on \changed{LILYGO T3-S3 V1.2 boards (ESP32-S3 with SX1280 2.4\,GHz LoRa)}. The architecture decouples the control and data planes: a LoRa radio serves exclusively as a low-power control plane, while sensor nodes use timer-based deep sleep ($<$10\,$\mu$A) with 180-second wake intervals to minimise idle power consumption. \changed{The gateway runs a persistent Wi-Fi access point; leaf nodes connect as STA clients on timer wake, announce their presence via CoAP POST, and serve images for download.} High-bandwidth payloads, simulated via pre-loaded images on SD cards, are segmented and \changed{reliably transmitted over the Wi-Fi data plane} using CoAP block-wise transfer over UDP. \changed{FreeRTOS dual-core task separation eliminates LoRa blackouts during harvest, AODV reactive mesh routing provides multi-hop coordination, and a Bully election algorithm enables nodes to self-organise into gateway, relay, and leaf roles without manual configuration.} By shifting from a continuous-listening network to an event-driven hybrid model, the system substantially reduces idle power consumption while retaining the capacity to route large files across multiple hops. The prototype demonstrates a scalable, energy-efficient framework capable of delivering visual alerts to a local gateway, shifting remote forest monitoring from reactive manual data retrieval to proactive intervention.

\section{Introduction, Problem Statement and Objectives}
% (Word Limit: 750 words)

\subsection{Introduction}

Effective environmental monitoring and forest conservation is critical for biodiversity preservation and disaster mitigation, which require sensor networks capable of operating autonomously in remote, off-grid environments. Currently, the industry relies heavily on manual data retrieval from local storage [1], which is a labor-intensive and reactive approach. This project proposes a Hybrid IoT Mesh Network utilizing ESP32-S3 microcontrollers to bridge the gap between remote sensing and real-time data access. By integrating LoRa for long-range, low-power control signaling and Wi-Fi for high-bandwidth data transfer, the system offers a highly scalable solution. \changed{To ensure rigorous and deterministic validation of the underlying network routing, this iteration of the project replaces live camera hardware with an SD card containing pre-loaded images. This approach utilizes images with various resolutions pre-loaded onto SD cards, eliminating camera hardware-induced errors and isolating the network's performance metrics for more accurate testing.} The system employs periodic wake cycles combined with event-driven coordination, enabling all high-bandwidth communications to utilize CoAP over UDP for lightweight, reliable image transmission via block-wise transfer. This architecture shifts remote monitoring from reactive data retrieval to proactive intervention.

\subsection{Problem Statement}

The primary challenge in off-grid environmental monitoring is the ``Data Latency Gap.'' Standard methods store data locally on SD cards, resulting in weeks or months of delay between an event occurring and its detection. While real-time wireless alternatives exist, they face a strict ``Bandwidth-Range-Power Trilemma'': low-power wide-area technologies (like LoRa) lack the bandwidth required for image transfer, while high-bandwidth solutions (like standard Wi-Fi or cellular networks) consume excessive energy and lack coverage in deep forests.

This project directly addresses this gap by decoupling the control and data transport layers, solving the power constraints that traditionally limit the long-term deployment of visual sensors. \changed{Furthermore, traditional embedded vision nodes frequently suffer from heap memory exhaustion when processing large image buffers in real-time. By implementing a storage-driven data pipeline that streams data directly from an SD card to the Wi-Fi buffer in 1024-byte chunks, this project circumvents the RAM limitations of the ESP32-S3. Additionally, the network design has been improved to operate completely independently of commercial LoRaWAN or 4G infrastructure; the gateway is now implemented purely as a designated ``Root Node'' within the network, bridging directly to a local diagnostic server for highly localized, off-grid deployment.}

\subsection{Objectives}

The main goal of this project is to develop and validate a functional prototype of a hybrid sensor node that overcomes the limitations of single-protocol networks. The specific objectives are categorized into implemented features and future work:

\subsubsection{Implemented Objectives}

\begin{enumerate}
    \item \textbf{Design a Hybrid Network Architecture:} Develop a sensor node utilizing the ESP32-S3 that integrates both LoRa \changed{(SX1280)} and Wi-Fi radios to optimize the trade-off between transmission range and data bandwidth.

    \item \textbf{Implement Low-Power Sleep Management:} Engineer a highly efficient power management system using timer-based deep sleep ($<$10\,$\mu$A) with 180-second wake intervals. \changed{Note: The originally planned LoRa DIO1 hardware interrupt wake mechanism is non-functional on the LILYGO T3-S3 V1.2 hardware; timer-based wake serves as the primary implementation.}

    \item \textbf{Optimize Image Transfer:} Implement CoAP over UDP with Block-Wise Transfer mechanisms to enable the reliable transmission of large payloads over a lossy, multi-hop Wi-Fi mesh.

    \item \textbf{Develop Dynamic Role Assignment:} Implement a flexible role configuration system allowing all ESP32-S3 nodes to be assigned as Leaf, Relay, or Gateway at deployment, with persistence across power cycles. \changed{Extend this with automatic role negotiation via a Bully election algorithm, enabling nodes to self-organize without manual configuration.}

    \item \changed{\textbf{Develop a File Reading Pipeline:} Design an SD-card-based file I/O system that simulates physical camera captures by randomly reading and segmenting pre-loaded JPEG files into network-safe chunks, ensuring stable payload generation without memory allocation failures.}

    \item \textbf{Evaluate System Performance:} \changed{Functionally validate the prototype through laboratory integration testing against its core objectives. Quantitative measurement of idle power consumption, throughput in kB/s, and end-to-end packet delivery ratio remains deferred to future work.}
\end{enumerate}

\subsubsection{Future Work}

The following objectives remain as planned enhancements beyond the current implementation scope:

\begin{enumerate}
    \item \textbf{Implement Queue-Based Scheduling:} Design a gateway-managed transmission queue that assigns time-division slots to prevent network congestion, ensuring only one node transmits image data at a time. \changed{Current implementation uses sequential harvest without explicit time-slot allocation.}

    \item \textbf{Security and Encryption:} Implement AES-128 encryption for LoRa control-plane packets to protect against eavesdropping and tampering. \changed{Current implementation transmits all packets in cleartext.}
\end{enumerate}

\section{Literature Review}
% (Word Limit: 500 words)

\subsection{Current State of Knowledge and Technology}

Environmental monitoring is transitioning from offline, manual camera traps to connected Wireless Sensor Networks (WSNs) [2], [3]. However, deployments remain fragmented by radio capability. High-bandwidth networks rely on 4G/LTE or standard Wi-Fi, which lack coverage in remote forests and require substantial power infrastructure [4]. \changed{LTE-M achieves 375\,kbps--1\,Mbps but draws 100--200\,mA during transmission [4], limiting unsolar-charged battery life to weeks.} Conversely, Low-Power Wide-Area Networks (LPWANs) such as LoRa provide multi-kilometre range (2--10\,km line-of-sight) with idle current $<$20\,$\mu$A, but are limited to $<$5\,kbps throughput and 222-byte payloads under duty-cycle restrictions [5].

\subsection{Significant Research Findings and Technological Developments}

Recent work has focused on routing protocols and energy management. LoRa sustains nodes for years with measured idle draw of 10--50\,$\mu$A in deep sleep [6]. \changed{AODV reactive routing enables infrastructure-less multi-hop coordination with 100--500\,ms route-discovery latency in 5--10 node networks [7].} Hybrid-network studies find that single-radio systems cannot simultaneously handle long-range idle listening and high-speed data bursts efficiently [8]. \changed{Hybrid LoRa+Wi-Fi architectures have reported order-of-magnitude energy savings over continuous Wi-Fi listening while retaining sufficient throughput for bulk payloads [8].}

\subsection{Common Methodologies and Their Effectiveness}

Previous studies commonly use single-radio duty-cycling, where a high-bandwidth node wakes on a fixed schedule [9]. \changed{Field deployments report 3--6 month battery life at 15-minute wake intervals, with 7.5-minute average alert latency and 200--400\,mA peak wake current [9]}, extending battery life at the cost of real-time responsiveness because the network is deaf between cycles. A second approach transmits compressed image data exclusively over LoRa; \changed{pure-LoRa image transmission requires 120--300\,s per 50\,kB JPEG at SF7 [10]}, causing high collision rates and rapid battery depletion when multiple nodes transmit simultaneously.

\subsection{Limitations and Shortcomings of Existing Solutions}

The primary limitation is the ``Bandwidth-Range-Power Trilemma'': high-bandwidth radios draw 80--200\,mA continuous idle, depleting 2000\,mAh batteries in 10--25 hours, whereas LoRa's sub-5\,kbps throughput cannot carry media files. Additionally, ESP32-class platforms (320\,kB SRAM) frequently exhaust the heap when buffering 50--200\,kB images alongside network stacks, leading to memory fragmentation and crashes reported in the embedded-vision literature [11].

\subsection{Relation of Previous Research to the Objectives of this Study}

Together, these findings motivate this project's dual-radio, event-driven architecture: LoRa for sub-100-byte control traffic to avoid image-transfer congestion [10], Wi-Fi with CoAP Block-Wise Transfer for bulk payloads to bypass LoRa duty-cycle limits, deep sleep to approach LPWAN idle currents [6], and SD-to-network streaming in 1024-byte chunks via CoAP (RFC 7959) to sidestep the heap-exhaustion failure mode observed in RAM-buffered designs [11].

\section{Design}
% (Word Limit: 750 words)

\subsection{Overall System Architecture}

The system uses a \changed{star-mesh hybrid} architecture with two operational planes:

{\color{red}
\begin{itemize}
    \item \textbf{Control Plane (LoRa, mesh topology):} All nodes broadcast beacons and routing packets over LoRa 2.4\,GHz. AODV reactive routing discovers multi-hop paths, and a Bully election selects the gateway. This plane handles node discovery, route management, election, and relay coordination.
    \item \textbf{Data Plane (Wi-Fi, star topology):} The gateway runs a persistent Wi-Fi access point; leaf nodes connect as STA clients and transfer images via CoAP. For out-of-range leaves, a relay performs sequential store-and-forward between the leaf's AP and the gateway's AP.
\end{itemize}
The LoRa mesh handles low-power coordination while the Wi-Fi star handles high-throughput data transfer, combining mesh resilience for control with star simplicity for bulk data.
}

\subsection{Hardware and Devices}

All nodes use identical hardware, differing only in runtime role (Leaf, Relay, or Gateway):

\begin{enumerate}
    \item \textbf{Microcontroller:} Espressif ESP32-S3.
    \item \textbf{Transceiver:} \changed{SX1280 2.4\,GHz LoRa with integrated PA on the LILYGO T3-S3 V1.2 board, connected via FSPI with dedicated RXEN/TXEN GPIOs.}
    \item \textbf{Display:} SSD1306 128$\times$64 OLED via I2C for real-time status (role, IP, image count, statistics).
    \item \textbf{File storage:} MicroSD card via HSPI. Camera hardware is simulated by pre-loading JPEG images onto the SD card for deterministic testing.
\end{enumerate}

\subsection{Network Topology and Data Flow}

\changed{During the idle phase, all nodes communicate via LoRa only. The gateway runs a persistent Wi-Fi AP named \texttt{ForestCam-GW-XXYY}. On timer wake, a leaf restores its role and cached gateway SSID from RTC memory, connects as STA to the gateway's AP, sends a CoAP \texttt{POST /announce} containing its MAC and image count, and serves its images for download via CoAP Block2 pipelined transfer before returning to sleep.} Out-of-range leaves are harvested by a relay via sequential store-and-forward between the leaf's and gateway's APs using a \texttt{HARVEST\_CMD}/\texttt{HARVEST\_ACK} protocol.

\subsection{Communication Protocols and Data Pipeline}

\begin{enumerate}
    \item \textbf{Control plane:} \changed{LoRa 2.4\,GHz carries 15-byte v2 beacons (every 30\,s), AODV RREQ/RREP/RERR packets, election packets, and harvest commands. A 3-byte magic header (\texttt{0xFC 0x01 type}) identifies ForestCam traffic.}
    \item \textbf{Transport:} CoAP (RFC 7252) over UDP on the Wi-Fi data plane.
    \item \textbf{Bulk transfer:} CoAP Block-Wise Transfer (RFC 7959) with 1024-byte blocks fits within the 1500-byte Wi-Fi MTU to prevent IP fragmentation. The ESP32-S3 streams JPEG data directly from the SD card into the Wi-Fi buffer in 1024-byte chunks, avoiding heap exhaustion. \changed{Pipelined requests with a window of 3 and a 2\,s timeout further improve throughput.}
\end{enumerate}

\subsection{Data Storage and Processing}

Raw data resides permanently on each leaf's local SD card; harvesting is a copy operation, never a move. \changed{The gateway saves harvested images to \texttt{/received/} with filenames encoding node MAC suffix, boot count, and uptime (e.g., \texttt{node\_AABB\_boot005\_003672s\_img\_000.jpg}), preventing collisions across sleep cycles.} Heavy processing such as decoding, compression, or full-file buffering is avoided on-device to prevent heap exhaustion.

\subsection{User Interaction and Integration}

The system avoids third-party cloud platforms because it targets fully off-grid operation. Three mechanisms are provided:

\begin{enumerate}
    \item \textbf{Local SD access:} harvested images are retrieved via physical SD removal or USB serial file transfer from the gateway's \texttt{/received/}.
    \item \textbf{Serial command interface:} runtime commands (\texttt{block}, \texttt{unblock}, \texttt{list}) control which nodes participate in harvest cycles without reflashing.
    \item \textbf{OLED status display:} each node shows role, IP, image count, and statistics for rapid field diagnosis.
\end{enumerate}

\subsection{Security Protocols}

The prototype implements baseline data-plane security. The gateway's Wi-Fi AP uses WPA2-PSK with a static pre-shared key, preventing unauthorised association and interception of CoAP transfers. All LoRa control packets carry the 3-byte magic header (\texttt{0xFC 0x01 type}) to distinguish ForestCam traffic from environmental noise, though this provides no cryptographic authentication. LoRa control packets remain in cleartext; AES-128 encryption and HMAC-based mutual authentication are deferred to future work (see Section~VI).

\subsection{Deployment Strategy, Challenges, and Mitigation}

\changed{The final build diverges from the proposal in three areas detailed below: AODV over LoRa replaces ESP-Mesh-Lite (Challenge~1), RTC timer wake replaces DIO1 interrupt wake (Challenge~2), and the gateway runs a persistent Wi-Fi AP (Challenge~3).} The prototype is bench-tested in a laboratory to verify routing and protocol operation; field testing remains future work.

\textbf{Challenge~1: SPI bus and mesh protocol choice.} \changed{Both the LoRa transceiver and the MicroSD card need SPI; the LILYGO T3-S3 V1.2 places them on separate hardware SPI buses, eliminating contention. The originally proposed ESP-Mesh-Lite was rejected because it requires all nodes to remain continuously awake, incompatible with the $<$10\,$\mu$A deep-sleep target; AODV over LoRa was adopted instead.}

\textbf{Challenge~2: LoRa blackout during harvest and unreliable DIO1 wake.} \changed{The original single-threaded \texttt{loop()} blocked LoRa reception for 50--120\,s during Wi-Fi harvest, solved by pinning LoRa/AODV/election to Core~0 and Wi-Fi/CoAP harvest to Core~1 via FreeRTOS. Separately, DIO1 interrupt wake from deep sleep proved unreliable on the LILYGO T3-S3 V1.2 (the SX1280 loses state via GPIO float during sleep), so the system uses RTC timer wake at 180\,s intervals.} CoAP application-layer retries (3 attempts, 2\,s timeout) compensate for UDP packet loss.

\textbf{Challenge~3: LoRa/Wi-Fi range mismatch (partially mitigated).} \changed{The SX1280 with PA reaches significantly further than ESP32-S3 Wi-Fi at 8.5\,dBm, so a leaf can be LoRa-reachable but Wi-Fi-unreachable because AODV routes reflect LoRa reachability only. Manual relay assignment via \texttt{HARVEST\_CMD} handles known out-of-range cases; automatic RSSI-based relay promotion remains future work.}

\subsection{Testing Scope and Known Gaps}

Testing combines 152 unit tests across 17 suites (election, harvest FSM, CoAP Block2 pipelining, beacon serialisation, node registry, relay harvest, AODV route expiry, deep sleep guards, thread safety) with 3-node integration tests (1~Gateway, 1~Relay, 1~Leaf) under laboratory conditions. \changed{Auto-role negotiation allows all nodes to boot as LEAF; the highest-priority node promotes to gateway after the 15\,s grace period, and image transfer is validated via both direct Wi-Fi and relay store-and-forward paths.} Fletcher-16 checksum verification confirmed end-to-end integrity with no observed corruption across multiple sessions.

Deliberate gaps remain: (i)~multimeter-based power measurement was not conducted (the $<$10\,$\mu$A idle figure is taken from ESP32-S3 and SX1280 datasheets); (ii)~no security penetration testing or WPA2-PSK key-strength validation; (iii)~no scalability testing beyond 3 nodes or 2 hops; (iv)~no outdoor, foliage, or temperature field validation; and (v)~no sustained stress testing of simultaneous leaf wake-ups or high-throughput multi-image cycles. Section~V reports the verified outcomes; Section~VI carries forward the unimplemented work.

\section{Results and Analysis}
% (Word Limit: 750 words)

\changed{Note: Sections~5.2--5.5 combine verified laboratory observations with forecasted projections; forecasted items are explicitly marked.}

\subsection{Verified Implementation Scope}

All core features described in Sections~III--IV compile, pass their unit tests, and run end-to-end in the 3-node laboratory topology. \changed{Verified outcomes include the hybrid LoRa--Wi-Fi data pipeline, CoAP Block2 transfer (50\,kB--5\,MB), FreeRTOS dual-core task separation (eliminating 50--120\,s LoRa blackouts during harvest), auto-role negotiation via Bully election, relay store-and-forward, beacon-reactive harvest, and Fletcher-16 integrity verification.} AES-128 LoRa encryption, TDMA scheduling, and the Python gateway dashboard remain unimplemented (see Section~VI).

\subsection{Scalability and Network Performance}

\textbf{[Forecasted, not validated in testing]} The architecture supports up to 8 discovered nodes in the NodeRegistry and 12 AODV routes (3-hop maximum depth). New leaves are auto-discovered via LoRa beacons with no firmware changes required. \changed{Only 3-node topologies (1 gateway + 1 relay + 1 leaf) were actually tested; multi-hop chains beyond 2 hops, concurrent relay operations, and deployments exceeding 3 nodes remain unvalidated.}

\subsection{Key Performance Metrics}

\textbf{[Verified, laboratory]} CoAP Block-Wise Transfer (1024-byte blocks, 3-retry limit, 2\,s per-block timeout) reliably delivered JPEG images from 50\,kB to 5\,MB via both direct Wi-Fi (1-hop) and relay store-and-forward (2-hop) paths. Fletcher-16 checksum verification confirmed end-to-end integrity with no observed corruption across multiple transfer sessions. Auto-role negotiation via Bully election completed within the 15--30\,s range defined by \texttt{ELECTION\_STARTUP\_GRACE\_MS} plus backoff, and gateway failover successfully triggered re-election when the acting gateway was powered off. \changed{Absolute throughput in kB/s and quantitative packet delivery ratio were not benchmarked.}

\textbf{[Estimated, datasheet]} \changed{Deep-sleep idle current is estimated at $<$10\,$\mu$A from the ESP32-S3 and SX1280 datasheets, and active draw (Wi-Fi + LoRa + SD) at 150--250\,mA; multimeter validation and battery-lifetime projection remain future work.}

\subsection{Comparison with Similar IoT Solutions}

\textbf{[Theoretical, no empirical benchmarking]} Compared to pure-LoRa image transmission [10] (120--300\,s per 50\,kB image at SF7), the hybrid design frees the LoRa channel for control signalling and delegates bulk transfer to Wi-Fi. Compared to always-on Wi-Fi meshes, the deep-sleep architecture is expected to reduce idle current by several orders of magnitude based on datasheet comparison. \changed{No side-by-side empirical benchmarking was performed.}

\subsection{Alignment with Initial Objectives}

Objectives~1, 3, 4, and 6 (hybrid architecture, CoAP Block2 image transfer, dynamic role assignment, SD file reading pipeline) are fully achieved. Objective~2 (wake-on-radio) is partially achieved using timer-based wake after DIO1 interrupt wake proved unreliable on the LILYGO T3-S3 V1.2. Objectives~5 (TDMA scheduling) and~7 (quantitative performance evaluation) remain partial and are carried forward to Section~VI.

\subsection{Observed Issues and Security Posture}

\textbf{Bottlenecks.} \changed{With the persistent gateway AP eliminating Wi-Fi hopping, the primary bottleneck becomes the ESP32 SoftAP's approximately 4-client concurrent-STA limit combined with sequential per-node CoAP downloads. Simultaneously-waking leaves compete for association slots, and transfer latency scales with image count and size.} The 1024-byte chunked reading pipeline prevented heap exhaustion across continuous multi-image transfers.

\textbf{Unexpected behaviours.} Occasional Wi-Fi timeouts cause the harvest state machine to skip the current node and retry on the next cycle; LoRa beacon fading is managed by the 60\,s registry staleness window. \changed{A multi-gateway election race during 3-node testing was traced to three causes: LEAF nodes not processing received beacons for gateway detection, promoted nodes still beaconing as \texttt{NODE\_ROLE\_LEAF}, and simultaneous grace-period expiry on the half-duplex LoRa channel. Fixes: beacon processing moved outside the RELAY-only gate, promoted nodes now beacon as \texttt{NODE\_ROLE\_GATEWAY}, and a 0--5\,s MAC-based jitter staggers grace periods.} A \texttt{CoapClient::verifyChecksum()} buffer overflow and a blocking \texttt{delay()} call in \texttt{HarvestLoop::\_doConnect()} were also found and fixed during code review.

\textbf{Security posture.} WPA2-PSK with a static pre-shared key secures the Wi-Fi data plane. The LoRa control plane currently transmits in cleartext; AES-128 encryption and HMAC-based mutual authentication are deferred to future work. This posture is acceptable for laboratory validation but would require hardening before field deployment.

\section{Conclusion}
% (Word Limit: 250 words)

\subsection{Achieved Outcomes}

This project developed and validated a hybrid LoRa + Wi-Fi mesh prototype that decouples low-power control signalling from high-bandwidth data transfer on commodity ESP32-S3 hardware. The most important finding is that CoAP Block-Wise Transfer with 1024-byte blocks over Wi-Fi, combined with AODV multi-hop routing over LoRa, reliably transfers JPEG images up to 5\,MB across a 3-node laboratory topology without heap exhaustion. Verified capabilities include auto-role negotiation via Bully election, timer-based deep sleep with RTC state persistence, relay store-and-forward, Fletcher-16 end-to-end integrity checking, FreeRTOS dual-core task separation (eliminating LoRa blackouts during harvest), and beacon-reactive harvest triggering. The prototype demonstrates that commodity hardware can perform reliable multi-hop file transfer in infrastructure-less environments, providing a foundation for remote environmental monitoring in regions without cellular connectivity.

\subsection{Limitations and Future Work}

Deployment readiness requires: (i)~security hardening (AES-128 LoRa encryption, penetration testing); (ii)~multimeter validation of the $<$10\,$\mu$A idle-current estimate and battery-life projection; (iii)~scalability testing beyond 3 nodes and 2 hops; (iv)~outdoor testing including foliage attenuation and temperature extremes; (v)~the planned Python gateway dashboard; and (vi)~self-healing relay fallback when direct Wi-Fi connections fail. Validating these items would transition the current laboratory prototype into a deployable wildlife-monitoring solution.
